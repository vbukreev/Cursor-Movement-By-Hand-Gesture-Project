## Purpose

This working agreement explains how our team will collaborate during Capstone II. It defines our meeting schedule, communication rules, accountability expectations, GitHub workflow, testing process, and deadline expectations.

---

## 1. Meeting Information

### Weekly Meeting

- **Meeting Day:** Thursday
- **Meeting Time:** 3:00 PM
- **Meeting Platform/Location:** In-person / WhatsApp

### Meeting Expectations

- Each member should attend weekly meetings or inform the team in advance if they cannot attend.
- Each member should give a short update on their assigned work.
- Meetings will be used to discuss progress, blockers, GitHub issues, and upcoming deadlines.

---

## 2. Communication Expectations

### Preferred Communication Channels

- WhatsApp group chat for quick updates and reminders
- GitHub Issues for task-specific updates
- Calls or in-person meetings for longer or urgent discussions

### Response Time

- Team members should respond to important messages within **24 hours**.
- If there is an urgent deadline, members should try to respond as soon as possible.
- If a member is blocked or cannot complete a task, they should inform the team early.

### Professional Conduct

All members are expected to:

- Be respectful during discussions
- Avoid hostile or dismissive language or behaviour
- Accept constructive criticism professionally
- Help maintain a collaborative environment

---

## 3. Accountability Rules

Each team member is responsible for completing assigned work and keeping the team updated.

### Missing Meetings

- If a member misses a meeting, they should review the meeting updates and follow up with the team.
- Repeated missed meetings may result in extra documentation or testing duties.

### Repeated Lateness

- If a member is repeatedly late, the team may reduce their role in project decisions until they are caught up.
- The member may also be asked to help with extra documentation or testing.

### Lack of Participation

- If a member is not contributing, the team will first discuss the concern directly with them.
- If the issue continues, responsibilities may be reassigned.
- If needed, the concern may be reported to the professor.

### Failure to Respond

- If a member does not respond to important messages within a reasonable time, the team may continue with the project decision if a deadline is close.
- Repeated failure to respond may be reported to the professor.

---

## 4. GitHub Workflow Agreement

GitHub will be used for code, documentation, issues, milestones, and project tracking.

### Branching Rules

- Direct pushes to `main` are not allowed for major code changes.
- Small documentation updates may be pushed directly to `main`.
- Feature work should be done on a separate branch.

### Branch Naming Convention

Branches should use short and clear names.

Examples:

- `feature/dashboard-ui`
- `feature/pinch-threshold`
- `fix/cursor-smoothing`
- `docs/update-readme`
- `test/gesture-testing`

### Pull Request Rules

- Code changes should be merged into `main` through a Pull Request.
- At least one team member should review the Pull Request before merging.
- The Pull Request should explain what was changed and which issue it connects to.
- Merge conflicts should be fixed before approval.

### Merging into Main

- Code should only be merged into `main` after review and basic testing.
- The `main` branch should stay stable and ready for demo.
- Broken or incomplete features should not be merged into `main`.

---

## 5. Testing Requirements

Testing is important because AeroPoint depends on webcam input, hand tracking, and real-time gesture control.

### Minimum Testing Before Merge

Before merging code, the assigned member should check that:

- The application runs without crashing
- The camera opens correctly
- Hand tracking still works
- Cursor movement works as expected
- Click and scroll gestures are not broken
- Any new feature is tested manually

### Automated Tests

- Automated tests are not required for every feature at this stage.
- If a feature can be tested with simple unit tests, the team may add them.
- Gesture logic and calibration logic should be tested where possible using sample values.

### Pull Request Verification

Before approving a Pull Request, the reviewer should check that:

- The code is readable
- The change matches the related issue
- The feature was tested
- No important files were accidentally changed
- The project still runs after the change

---

## 6. Deliverables and Deadlines

### Early Submission Requirements

- Deliverables should be completed at least **24 hours before the deadline** when possible.
- This gives the team time for review, testing, bug fixes, documentation updates, and peer suggestions.
- If a member cannot complete their assigned work on time, they should inform the team as early as possible.

## 7. Agreement

By following this working agreement, our team will stay organized, communicate clearly, and complete the AeroPoint Capstone II project in a fair and professional way.
